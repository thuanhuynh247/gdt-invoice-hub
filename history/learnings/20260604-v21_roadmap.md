## 1. Architectural Lessons & Takeaways

### A. Graph Analytics for Tax Fraud Detection
- **Lesson:** Standard relational queries fail to identify complex circular invoicing rings (fake companies buying and selling among themselves to inflate costs/claim illegal VAT refunds). Building directed buyer-seller network graphs and implementing depth-first cycle detection (limiting search depth to 5 to avoid exponential scaling) makes shell rings highly visible.
- **Pattern:** Using HITS (Hubs and Authorities) scoring identifies central invoice distributors (high Hub score) and massive cost-sink companies (high Authority score), which is the exact pattern of systemic VAT fraud networks.

### B. Sequential Cryptographic Ledgers for Tamper Evidence
- **Lesson:** To prevent retroactive altering of database invoices (a common tax evasion technique), transactions must be recorded sequentially in an immutable ledger backed by a Merkle Tree. Re-verifying the Merkle root on every ledger audit guarantees data integrity.
- **Pattern:** Zero-Knowledge Proofs (ZKP) can verify compliance bounds (e.g., that every transaction's VAT rate complies with decree limits) without leaking individual monetary amounts.

## 2. Technical Decisions & Refinements

### A. Customs XML Ingestion & Matching
- **HS Code and VAT Matching:** VNACCS Customs XML declarations contain complex nested tags. Compiling tax bases and HS codes, converting foreign currencies via specific daily exchange rates (e.g., VCB rates), and matching against input VAT invoices ensures that custom duties match domestic declarations.
- **Mitigation Proposals:** When variance is detected, providing automated correction proposals (e.g., adjusting VAT base or customs duty declarations) helps tax accountants clear customs audits quickly.

## 3. Critical Patterns Promoted

1. **Cycle Detection in directed buyer-seller graphs:**
   - Always bound the cycle search depth (e.g., depth limit of 5) to ensure high performance on large transaction graphs.
2. **Immutable Merkle Ledger Pattern:**
   - Link each newly added invoice to the previous transaction hash and update the Merkle tree root to guarantee chronological consistency.
